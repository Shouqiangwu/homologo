# 网络安全（R155）与软件更新（R156）要求

> UNECE R155 CSMS + UNECE R156 SUMS

---

## 概述

R155和R156是**组织级+车型级**双层认证，不仅要求单个车型合规，还要求OEM的管理体系通过审核。对中国车企而言，这是进入EU市场的关键门槛，因为需要在组织层面建立并维护网络安全和软件更新的管理体系。

## 适用范围和时间表

| 法规 | 适用类别 | 新车型强制 | 所有新车强制 | 状态 |
|------|---------|-----------|-------------|------|
| R155 | 全部M/N | 2022-07-06 | 2024-07-07 | ✅ 已完全强制 |
| R156 | 全部M/N | 2022-07-06 | 2024-07-07 | ✅ 已完全强制 |

## R155 网络安全管理系统（CSMS）

### 双层认证结构

| 层级 | 对象 | 有效期 | 审核机构 |
|------|------|--------|---------|
| **CSMS证书** | OEM组织（整个公司） | 3年，可续期 | 型式认证机构（如KBA, RDW, e2等） |
| **车型网络安全型式认证** | 具体车型 | 随CSMS证书有效期 | 同上 |

**→ 关键理解：** 没有CSMS证书，无法获得任何车型的网络安全型式认证。CSMS是前提条件。

### CSMS要求 — R155 §7.2

OEM必须证明其CSMS覆盖以下方面：

| 要求领域 | 具体要求 |
|---------|---------|
| 威胁识别与风险评估 | 识别车辆系统面临的网络威胁，评估风险等级 |
| 设计阶段安全 | 在车辆开发阶段将网络安全纳入设计流程（security by design） |
| 供应链管理 | 管理供应商的网络安全要求和监控 |
| 事件检测与响应 | 建立网络安全事件的检测、报告和响应机制 |
| 持续监控 | 在车辆全生命周期内持续监控网络安全威胁 |
| 更新能力 | 具备在发现漏洞后及时修复的能力 |

### 车型网络安全要求 — R155 §7.3

| 要求领域 | 具体要求 |
|---------|---------|
| 风险评估文档 | 提供车型特定的威胁分析和风险评估（TARA） |
| 缓解措施 | 针对识别的威胁实施了适当的缓解措施 |
| 验证测试 | 通过测试验证缓解措施的有效性 |
| 持续监控计划 | 车型上市后的网络安全监控计划 |
| 事件响应计划 | 车型特定的网络安全事件响应流程 |

### R155 Annex 5 — 威胁类别和缓解措施清单

R155 Annex 5列出了车辆面临的威胁类别和对应的缓解措施示例。OEM在TARA中必须逐一评估：

| 威胁类别 | 示例 |
|---------|------|
| 后端服务器威胁 | 未授权访问车辆数据；DoS攻击 |
| 通信信道威胁 | V2X通信伪造；OTA更新劫持 |
| 更新程序威胁 | 恶意软件通过OTA注入 |
| 人为失误 | 开发阶段遗留的调试端口 |
| 外部连接威胁 | 通过OBD-II端口、USB、蓝牙等接口的攻击 |
| 车辆数据/代码威胁 | 固件提取、逆向工程、数据篡改 |

## R156 软件更新管理系统（SUMS）

### 双层认证结构（同R155）

| 层级 | 对象 | 有效期 |
|------|------|--------|
| **SUMS证书** | OEM组织 | 3年，可续期 |
| **车型软件更新型式认证** | 具体车型的OTA能力 | 随SUMS证书 |

### SUMS要求 — R156 §7.1

| 要求领域 | 具体要求 |
|---------|---------|
| 更新管理流程 | 记录、管理和追踪所有软件更新 |
| 更新影响评估 | 评估每次更新是否影响型式认证合规性 |
| 更新完整性 | 确保更新包的完整性和真实性（签名/加密） |
| 更新失败处理 | 更新失败时车辆回退到安全状态 |
| 更新记录 | 维护软件版本和更新历史记录 |

### OTA更新的特殊要求 — R156 §7.2

| 要求项 | 具体要求 |
|--------|---------|
| 用户通知 | 在OTA更新前通知用户，告知更新内容和预计时间 |
| 用户同意 | 安全相关更新需要用户确认（部分情况下可强制） |
| 更新时机 | 不得在驾驶过程中执行影响驾驶安全的更新 |
| 回退能力 | 如更新失败，车辆必须能恢复到可用状态 |
| RXSWIN | 每次影响型式认证的更新必须更新RXSWIN（Regulated Software Identification Number） |

### RXSWIN — 法规软件版本标识

| 要求项 | 说明 |
|--------|------|
| 定义 | RXSWIN是标识受法规约束的软件版本的唯一编号 |
| 适用场景 | 当软件更新影响了型式认证范围内的系统（如AEB、ISA、ELKS等）时 |
| 报告义务 | OEM必须在更新RXSWIN后通知型式认证机构 |
| 影响评估 | 更新后需评估是否需要重新做型式认证测试 |

## 合规检查清单

### R155 网络安全
- [ ] OEM是否已获得CSMS证书？（组织级）
- [ ] 车型是否已获得网络安全型式认证？
- [ ] 是否提供了车型特定的TARA（威胁分析和风险评估）文档？
- [ ] TARA是否覆盖了Annex 5中所有相关威胁类别？
- [ ] 是否有供应链网络安全管理措施？
- [ ] 是否有持续监控计划（上市后）？
- [ ] 是否有网络安全事件响应流程？

### R156 软件更新
- [ ] OEM是否已获得SUMS证书？（组织级）
- [ ] 车型是否已获得软件更新型式认证？
- [ ] OTA更新是否在更新前通知用户？
- [ ] 安全相关更新是否获取用户确认？
- [ ] 是否有更新失败的回退机制？
- [ ] 是否定义了RXSWIN？
- [ ] 影响型式认证的更新是否更新了RXSWIN？
- [ ] 更新后是否通知型式认证机构？
- [ ] 是否维护了完整的软件版本和更新历史记录？

## 常见不合规情况

1. **无CSMS/SUMS组织级证书**：中国车企首次进入EU市场，需从零建立管理体系并通过审核。这通常需要6-12个月准备时间。
2. **CSMS仅覆盖中国团队**：CSMS必须覆盖EU市场运营相关的所有组织环节，包括EU的售后服务团队。
3. **OTA更新跳过型式认证影响评估**：中国车企习惯的高频OTA模式（每2-4周推送），每次更新是否影响ADAS系统的型式认证都需要评估。
4. **供应商网络安全未管理**：一级供应商（Tier 1）的网络安全措施未纳入CSMS范围。
5. **RXSWIN未更新**：安全相关软件更新后未更新RXSWIN，未通知认证机构。
6. **驾驶中OTA更新**：部分系统在驾驶过程中下载和安装非安全关键更新，但未明确告知用户。
