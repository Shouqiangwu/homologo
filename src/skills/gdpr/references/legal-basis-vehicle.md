# 车辆数据处理的合法性基础

> GDPR Art.6 + Art.9 + ePrivacy Art.5(3) + EDPB Guidelines 01/2020

---

## 双层合法性要求

车联网数据处理需满足**两层**合法性要求：

| 层级 | 法律依据 | 适用场景 |
|------|---------|---------|
| **第1层：ePrivacy Art.5(3)** | 存取车辆终端设备中信息前需获得同意（除非豁免） | 车辆被视为"终端设备"（如手机），任何读取/写入车载数据的操作都触发此条 |
| **第2层：GDPR Art.6** | 个人数据处理需有合法性基础 | 所有涉及个人数据的处理操作 |

**→ EDPB的关键立场：** 大多数车联网数据处理场景需要**同意**（consent）作为合法性基础。合同（contract）和合法利益（legitimate interest）仅在有限场景下适用。

## GDPR Art.6 合法性基础 — 车联网场景适用

| 合法性基础 | 适用场景 | EDPB立场 |
|-----------|---------|---------|
| **Art.6(1)(a) 同意** | 大多数车联网服务（远程诊断、行为数据分析、位置追踪、保险UBI等） | ✅ EDPB推荐的主要基础 |
| **Art.6(1)(b) 合同** | 用户明确请求的服务所必需的数据处理（如GPS导航服务） | ⚠️ 仅限"必要"的范围 |
| **Art.6(1)(c) 法律义务** | eCall紧急呼叫（EU 2015/758强制）、EDR数据记录 | ✅ 法律强制的场景 |
| **Art.6(1)(f) 合法利益** | OEM质量改进、安全研究 | ⚠️ EDPB态度谨慎，需通过利益平衡测试 |

### 同意的特殊要求（车联网场景）

| 要求项 | 具体要求 | EDPB Guidelines引用 |
|--------|---------|-------------------|
| 分别同意 | 每个处理目的单独获取同意，不得与购车合同捆绑 | §56-58 |
| 自由同意 | 拒绝同意不得影响车辆基本功能的使用 | §54 |
| 知情同意 | 必须充分告知数据处理的目的、范围、接收方 | §57 |
| 可撤回 | 随时可撤回同意，撤回方式必须与给予同意同样便捷 | §59 |
| 多数据主体 | 车辆有多个使用者（车主、家人、乘客），需考虑所有数据主体 | §87-89 |

## Art.9 特殊类别数据 — 车联网场景

| 数据类型 | Art.9类别 | 说明 |
|---------|---------|------|
| **DMS面部图像** | 生物识别数据 Art.9(1) | 用于识别驾驶员身份时构成生物识别数据 |
| **位置数据推断** | 可能揭示宗教/性取向等 Art.9(1) | EDPB认为位置数据可能推断出敏感信息（如经常去教堂→宗教信仰） |
| **健康相关数据** | 健康数据 Art.9(1) | DDAW检测到的疲劳状态可能构成健康数据 |

**→ Art.9处理基础：** 在车联网场景下，**明确同意**（explicit consent, Art.9(2)(a)）几乎是唯一可行的Art.9处理基础。

## DPIA（数据保护影响评估）触发条件

EDPB认为以下车联网场景**必须**进行DPIA（Art.35）：

| 触发条件 | 车联网示例 |
|---------|---------|
| 大规模处理特殊类别数据 | DMS大规模收集生物识别数据 |
| 系统性监控公共区域 | 外部摄像头持续拍摄道路环境（含行人面部） |
| 位置数据大规模追踪 | 车辆GPS轨迹持续上传云端 |
| 自动化决策 | 基于驾驶行为数据的保险定价（UBI） |

## 车端本地处理 vs 云端处理

EDPB强烈推荐**车端本地处理**（local/in-vehicle processing）：

| | 车端本地处理 | 云端处理 |
|--|------------|---------|
| EDPB立场 | 推荐 | 需额外论证必要性 |
| GDPR适用 | 可能落入"家庭豁免"（Art.2(2)），不适用GDPR | 完全适用GDPR |
| 隐私风险 | 低 — 数据不离开车辆 | 高 — 网络传输+云存储风险 |
| 典型场景 | DMS面部识别在车端完成，不上传 | 远程诊断数据上传OEM服务器 |

**→ 关键规则：** 只要数据不离开车辆传输给任何控制者，处理可能不受GDPR约束。一旦数据传出车辆，GDPR完全适用。

## 合规检查清单

- [ ] 是否明确了每项数据处理的GDPR Art.6合法性基础？
- [ ] 同意是否与购车合同/服务条款分别获取？
- [ ] 拒绝同意是否不影响车辆基本功能？
- [ ] 是否处理了Art.9特殊类别数据（DMS、位置推断、健康数据）？
- [ ] 如处理Art.9数据，是否获取了明确同意？
- [ ] 是否进行了DPIA？
- [ ] 是否优先采用车端本地处理而非云端处理？
- [ ] 如有云端处理，是否论证了其必要性？
- [ ] 是否为多数据主体（车主/驾驶员/乘客）设计了同意管理机制？

## 常见不合规情况

1. **同意与购车合同捆绑**：在购车合同中嵌入数据处理同意条款，违反"自由同意"要求。
2. **DMS数据处理无Art.9基础**：未识别DMS面部图像为生物识别数据，未获取明确同意。
3. **无DPIA**：大规模GPS追踪和DMS部署未进行数据保护影响评估。
4. **拒绝同意导致功能降级**：用户拒绝数据收集同意后，车辆的导航、语音等基本功能被限制。
5. **乘客数据未考虑**：仅考虑了车主/驾驶员的同意，未考虑乘客作为数据主体的权利。
