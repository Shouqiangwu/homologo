# R156 SUMS与ISC集成

> UNECE R156 + EU 2019/2144 + EU 2018/858 Art.52-54

---

## R156的ISC维度

UNECE R156要求OEM建立SUMS并证明：
1. 软件可被安全更新（型式认证要求）
2. 更新后的软件仍符合所有适用法规（ISC连续性）
3. 车辆全生命周期内软件状态可追溯

**与ISC的交集：** 任何OTA更新都可能改变车辆的在用性能特征，ISC测试需覆盖OTA后的状态。

## 双证书体系

OEM需要两份关键证书：

| 证书 | 范围 | 颁发条件 |
|------|------|---------|
| **SUMS Certificate** | OEM企业级的软件更新管理体系 | 企业流程审计通过 |
| **Vehicle Type SUMS-relevant Approval** | 特定车型的SUMS实施 | 车型软件架构符合SUMS |

两份证书配合使用：SUMS认证证明企业有能力管理更新，车型批准证明具体车型符合要求。

## 软件版本追溯

### 关键数据结构

```
VIN
 │
 ├─ 出厂软件版本清单
 │   ├─ ECU 1: software v1.2.3, RXSWIN-R155-001
 │   ├─ ECU 2: software v3.4.5, RXSWIN-R152-001
 │   └─ ...
 │
 ├─ OTA更新历史
 │   ├─ 2024-01-15: ECU 2 → v3.4.6 (Category B)
 │   ├─ 2024-05-20: ECU 1 → v1.3.0 (Category A) 
 │   └─ ...
 │
 └─ 当前软件状态
     ├─ 所有ECU当前版本
     └─ 当前RXSWIN集合
```

### 查询接口要求

| 接口 | 要求 |
|------|------|
| OBD诊断 | 必须能读取所有ECU版本 + RXSWIN |
| 远程查询（OEM） | 云端有VIN级软件状态数据库 |
| 认证机构查询 | 提供API或定期报告 |
| 市场监督 | 抽查时可读取 |

## ISC场景下的R156关联

### 场景1：ISC测试前的软件验证

```
车辆进入ISC测试
    ↓
读取车辆OBD → 获取当前软件版本和RXSWIN
    ↓
查询OEM数据库 → 该VIN应有的软件状态
    ↓
比对：
├─ 一致 → 正常ISC测试
├─ 不一致（客户未接受OTA）→ 按当前版本测试 + 记录
└─ 版本超前（未知版本）→ 调查异常
```

### 场景2：ISC失败，软件更新可修复

```
ISC测试失败（如NOx超标）
    ↓
OEM分析：是否可通过软件更新修复？
    ↓
├─ 是 → 开发更新 → 按R156 Category A流程 → 推送 → 重新ISC验证
└─ 否 → 硬件召回
```

### 场景3：OTA更新后的主动ISC

```
OTA更新推送完成（Category A）
    ↓
目标：验证更新后的实际性能
    ↓
OEM从接收更新的车辆中抽样
    ↓
执行ISC测试（排放 / ADAS）
    ↓
确认无退化 → 归档
    ├─ 若退化 → 紧急回滚 / 下一版本修复
    └─ 若改善 → 记录为成功案例
```

## 关键的时间协调

| 事件 | R156要求 | ISC要求 | 整合 |
|------|---------|---------|------|
| Category A OTA前 | 认证机构预审 | 预期性能不退化 | 同步提交合规证据 |
| OTA后0-30天 | 推送成功率监控 | 性能监控基线更新 | 数据共享 |
| OTA后3-6月 | 回滚决策窗口 | ISC抽样数据 | 综合决策 |
| 年度 | SUMS审计 | 排放ISC报告 | 合并汇报 |

## SUMS审计要点

### 审计内容（认证机构视角）

| 项目 | 审计问题 |
|------|---------|
| 变更决策记录 | 每次OTA的分类依据是什么？是否按R156执行？ |
| 测试证据 | Category A更新是否有完整测试报告？ |
| 认证机构通知 | Category B通知是否及时、完整？ |
| RXSWIN管理 | 版本追溯链是否完整？ |
| 回退案例 | 是否有回退失败或问题回退的记录？ |
| 用户通知 | 推送前通知用户的机制是否合规？ |
| 网络安全 | 与R155 CSMS的接口是否有效？ |

### 常见审计发现

1. Category分类过于保守（全部按C处理）
2. RXSWIN与实际车辆状态不一致
3. 第三方软件组件未纳入SUMS
4. 回退程序未实测
5. 用户同意管理不规范

## 对中国车企的挑战

### 挑战1：SUMS与中国开发流程脱节

| 中国习惯 | R156要求 | 冲突 |
|---------|---------|------|
| 快速迭代，频繁OTA | 严格分类，合规流程慢 | 产品节奏 vs 合规 |
| 灰度发布（A/B测试） | 同VIN同版本要求 | 实验性 vs 一致性 |
| 第三方组件灵活替换 | SUMS需覆盖全部 | 供应链管理 |
| 国内市场OTA自由度高 | EU市场严格 | 双标准维护 |

### 挑战2：云端数据基础设施

- EU数据合规（GDPR + data sovereignty）
- 需要EU数据中心
- VIN-软件状态查询需24/7可用

### 挑战3：认证机构沟通

- 语言：认证机构技术沟通英文/德文
- 文化：EU机构慢节奏，文档要求完整
- 响应：Category A更新前的预审需要提前6-8周

## 合规检查清单

- [ ] 是否持有R156 SUMS证书？
- [ ] 每个车型是否有对应的SUMS批准？
- [ ] VIN级软件状态数据库是否可查询？
- [ ] RXSWIN管理是否覆盖所有受规软件？
- [ ] Category A/B/C分类流程是否文档化？
- [ ] 认证机构是否有定期沟通机制？
- [ ] OTA前是否有影响评估？
- [ ] OTA后是否有ISC验证？
- [ ] 回滚程序是否测试过？
- [ ] 用户通知和同意机制是否合规？
- [ ] 第三方软件组件是否纳入SUMS？
- [ ] EU本地数据中心是否部署？

## 常见不合规情况

1. **SUMS证书覆盖不全**：企业SUMS有证，但新车型未单独走SUMS批准流程。
2. **灰度发布违反R156**：部分用户收到新版本，VIN-软件映射无法解释。
3. **RXSWIN不更新**：Category B更新后只更新内部版本号，忘记RXSWIN。
4. **ISC测试用老版本**：车辆已接受OTA但ISC测试未读取实际版本，数据混淆。
5. **Category A未预审就推送**：OEM自评估走Category B，但认证机构判定Category A。
6. **第三方组件漏洞**：导航、语音识别等第三方组件的更新未走SUMS。
7. **国内/EU版本混淆**：中国OTA推送与EU标准冲突，被EU市场监督发现。
8. **回滚能力纸面上有实际没**：SOP文档有但从未在真实场景测试。
